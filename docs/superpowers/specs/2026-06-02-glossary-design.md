# Health Benefits Glossary System — Design Spec

**Date:** 2026-06-02  
**Branch:** MH-keyword-help  
**Status:** Approved — ready for implementation

---

## Overview

A contextual glossary system that automatically detects health benefit terms in AI responses and UI text, bolds them with a help icon, and shows an inline popover with a plain-language explanation in the user's selected language.

---

## Goals

- Surface definitions for unfamiliar health/insurance terms without leaving the page
- Support all 6 app languages: `en`, `es`, `zh-CN`, `ht`, `pt-BR`, `vi`
- Work across chat responses, benefit descriptions, appeal summaries, and prescreener results
- Allow non-developer admins to add and edit terms over time

---

## Architecture

### Approach

**Client-side post-processing with a cached glossary index.**

The AI returns plain text. After rendering, a `TermScanner` utility scans the text for known glossary terms and splits it into annotated segments. A `<GlossaryText>` React component renders those segments — plain text passes through unchanged; matched terms render as bold text with a ⓘ icon. Clicking the icon opens an inline popover that fetches the full definition (with translation) on demand.

The glossary index is fetched once per session via SWR and cached for 1 hour. The per-term definition fetch happens only on first popover open and is memoized for the session.

## Data Model

### Supabase table: `glossary_terms`

```sql
CREATE TABLE glossary_terms (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug            text UNIQUE NOT NULL,       -- e.g. "deductible", "masshealth-standard"
  term_en         text NOT NULL,              -- canonical display name
  definition_en   text NOT NULL,              -- 2-4 sentence plain-English explanation
  definition_es   text,                       -- seeded
  definition_zh_cn text,                      -- AI-generated on demand, cached
  definition_ht   text,                       -- AI-generated on demand, cached
  definition_pt_br text,                      -- AI-generated on demand, cached
  definition_vi   text,                       -- AI-generated on demand, cached
  category        text NOT NULL,              -- "program" | "insurance" | "aca" | "medical"
  aliases         text[] DEFAULT '{}',        -- alternate forms for matching
  related_slugs   text[] DEFAULT '{}',        -- slugs of related terms
  created_at      timestamptz DEFAULT now(),
  updated_at      timestamptz DEFAULT now()
);

CREATE INDEX ON glossary_terms (category);
```

### Translation strategy

- **English + Spanish**: seeded in the migration file (`seed-terms.ts`)
- **zh-CN, ht, pt-BR, vi**: generated by Claude Haiku on first popover open for that language, then persisted to the DB. If generation fails, falls back to English.

## API Routes

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/glossary` | Returns all terms: `slug`, `term_en`, `aliases`, `category`. Cached `revalidate: 3600`. |
| GET | `/api/glossary/[slug]?lang=es` | Returns full term with definition in requested language. Generates + caches translation if missing. |
| POST | `/api/admin/glossary` | Create a new term (admin only) |
| PATCH | `/api/admin/glossary/[slug]` | Update a term (admin only) |
| DELETE | `/api/admin/glossary/[slug]` | Delete a term (admin only) |

## Scanner Logic

`scanner.ts` builds a single case-insensitive whole-word regex from all slugs and aliases (longest match first to avoid partial shadowing). Returns `Segment[]`:

```ts
type Segment =
  | { type: 'text'; content: string }
  | { type: 'term'; content: string; slug: string }
```

- Whole-word match only (`\b` boundaries) — won't match "copay" inside "copayment"
- Longest alias wins — "out-of-pocket maximum" matches before "out-of-pocket"
- Case-insensitive match, preserves original casing in rendered output
- Performance: 50-term regex against a typical chat response (~500 words) < 1ms

## Component API

```tsx
// Wrap any string — works in chat, static UI, anywhere
<GlossaryText text="Your deductible resets each plan year." />

// GlossaryTerm rendered internally:
// → "Your **deductible** ⓘ resets each plan year."
// → click ⓘ → popover with definition in user's language
```

**Popover content:**
- Term name (bold)
- Definition in user's selected language (fetched from `/api/glossary/[slug]?lang=`)
- Loading spinner during AI translation generation
- Up to 3 related term chips (clickable — opens their popover)
- "View all terms" link → `/knowledge-center#glossary` (future)
